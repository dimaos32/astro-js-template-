# Шаблон для статичексой верстки на Astro, SCSS и JS

> В шаблоне не используется `TypeScript` (несмотря на то, что с ним удобнее и быстрее работать с Astro) потому-что он предназначается для статической верстки, и с ним потом со стороны заказчика могут работать разработчики разного уровня подготовки, в том числе и не фронтендеры.

Для начала работы у вас должент быть установлен **Node.js 22 версии** или выше.

В шаблоне присутствуют конфиги `eslint`, `stylelint` и `prettier` - настоятельно рекомендуется установить соответствующие плагины в редактор кода.

Используется следующие postcss-плагины (настраиваются в `postcss.config.js`):

- `postcss-preset-env` - преобразует современный CSS в совместимый со старыми браузерами. Включает Autoprefixer для автоматической расстановки вендорных префиксов.
- `postcss-combine-media-query` - склеивает одинаковые медиазапросы в один в билде.
- `postcss-sort-media-queries` - сортирует медиазапросы в заданном порядке, направление в конфиге.
- `postcss-pxtorem` - преобразует все размеры в пикселях в rem, корневое значение - 16px, задается в конфиге.

Поддерживается следующий список браузеров:

```javascript
[
  'last 6 versions',
  'not dead',
  'not op_mini all',
  'Safari >= 15.4',
  'Firefox ESR',
];
```

> Все устройства Apple 2017 года выпуска и позже могут обновиться до версии 15.4.

В шаблоне используются современные CSS-свойства (например логические свойства). Если нужна поддержка более старых браузеров — измените настройки в browserslist файла package.json и проверьте стили.

## Структура проекта

Используется модульная архитектура

```text
├── plugins/           # кастомные плагины (спрайт, hmr для scss файлов и т.п.)
├── public/            # статические файлы (шрифты, изображения и т.п.)
├── src/
│   ├── components/
│   │   ├── common/    # обычные компоненты
│   │   ├── modules/   # крупные блоки - (типа хэдера, футера, секций),
│   │   │              # а так же сложные компоненты, могут иметь
│   │   │              # свою логику (например слайдеры)
│   │   └── ui/        # интерфейсные компоненты (кнопки, чекбоксы и т.п.)
│   ├── const/
│   ├── data/
│   ├── helpers/
│   ├── layouts/       # лэйауты страниц
│   ├── pages/         # страницы
│   ├── raw/           # исходники
│   │   ├── icons/     # svg-иконки, которые собираются в спрайт
│   │   └── images/    # неоптимизированные изображения, которые можно
│   │                  # оптимизировать встроенной утилитой
│   ├── scripts/
│   ├── settings/      # настройки для модулей и других сущностей
│   └── styles/        # глобальные стили, миксины, переменные и т.п.
├── utils/             # утилиты ( автоматизация процессов, генерация
│                      # компонентов, оптимизация картинок и тп. )
├── package.json
└── ...
```

Все статические файлы (шрифты, изображения и т.п.) могут лежать в папке `public/` - все файлы из нее автоматически попадают в билд.

## Команды

| Command               | Action                                            |
| :-------------------- | :------------------------------------------------ |
| `npm ci`              | Установить зависимости                            |
| `npm run dev`         | Запустить локальный дев сервер `localhost:4321`   |
| `npm run build`       | Собрать билд для продакшна `dist/`                |
| `npm run preview`     | Посмотреть билд локально перед деплоем            |
| `npm run gen:sprite`  | Собрать спрайт из файлов в папке `src/raw/icons/` |
| `npm run lint:js`     | Запустить линтер скриптов                         |
| `npm run lint:css`    | Запустить линтер стилей                           |
| `npm run lint`        | Запустить все литнеры                             |
| `npm run proc:images` | Утилита для оптимизации изображений               |

В шаблоне присутствует файл `package-lock.json`, и поэтому рекомендуется устанавливать зависимости современной командой `npm ci`.

Спрайт пересобирается каждый раз при изменении содержимого папки `src/raw/icons/` и при сборке билда - необходимость в ручном запуске возникает крайне редко.
Собранный спрайт помещается в файл `public/svg/sprite.svg`, при пересборке файл переписывается, если такой файл существует - поэтому необходимо следить самостоятельно, чтобы файла пользователя с таким именем не было.

При сборке билда сначала запускаются линтеры. Если есть ошибки - билд не соберется, предупреждения не влияют на сборку билда.

Скрипт `preview`превью запускает локальный билд для содержимого папки `dist/` (сам билд надо собирать отдельно). Так, как политики CORS становятся все строже и строже - при открытии html-файла без сервера (например из проводника) современные браузеры блокируют доступ почти ко всем ресурсам и ассетам.

Утилита `proc:images` оптимизирует изображения в папке `src/raw/images/` и помещает их в папку `public/images/`. Допустимы файлы форматов `jpg`, `jpeg` или `png`.

## Работа с изображениями

При запуске утилиты `proc:images` необходимо выбрать несколько настроек - очистить директорию `public/images/`, использовать ретинизацию, использовать WebP и качество изображений:

- Можно как очистить папку от ранее сгенерированных изображений, так и обновлять их свежими версиями (по умолчанию `No`). Очищать директорию бывает полезно, когда в исходниках были удалены или переименованы файлы. Старые файлы перезаписываются новыми - поэтому в большинстве случаев предпочтительнее очищать директорию, чтобы не пологаться на информацию из памяти - были или не были удалены или переименованны файлы. Если используется утилита - настоятельно **не рекомендуется** помещать самостоятельно файлы в папку `public/images/` во избежание коллизий.
- В случае оптимизации ретинизированной графики (по умолчанию `Yes`) исходные файлы считаются вариантом `2x`, в название файла добавляется постфикс (`@1x`, `@2x`)
- Генерация `WebP` опциональна (по умолчанию `Yes`).
- Качество изображений по-умолчанию установленно в `80%` и в большинстве случаев это оптимальный вариант.

> Компоненты `Astro Picture` и `Astro Iсon` не используются, т.к. в случае с `Picture` отсутствует возможность контролировать имена и расположение выходных ассетов, а в случае с `Icon` - `svg` либо инлайнинся, либо собирается в инлайн спрайт на странице (при многократном использовании на странице) - и то, и другое не подходит для статической верстки.

## Алиасы

В шаблоне используются алиасы для удобства импортов:

| Alias         | Path                      |
| :------------ | :------------------------ |
| `@`           | `/src`                    |
| `@ui`         | `/src/components/ui`      |
| `@components` | `/src/components/common`  |
| `@modules`    | `/src/components/modules` |
| `@styles`     | `/src/styles/global`      |

При необходимости можно добавить любые другие алиасы - в файл `astro.config.mjs` чтобы их понимал Astro и в `jsconfig.json` чтобы их понимал редактор кода.

## Детализированная структура проекта

### Компоненты

#### Общие компоненты (`src/components/common`)

Всевозможные компоненты, которые могут быть использованны любыми компонентами на разных уровнях.

#### Модули (`src/components/modules`)

Как правило это крупные блоки страниц - например хэдер, футер и секции. В идеале в шаблонах страниц должны использоваться только они, за редкими исключениями. Так же к модулям стоит относить крупные и или сложные (имеющие свою серьезную логику) компоненты - например модальное окно.

#### Интерфейсные компоненты (`src/components/ui`)

Элементы пользовательского интерфейса - кнопки, заголовки, кастомные реализации элементов формы (чекбоксы, радиобаты, выпадающие списки, инпуты и т.д.) и всевозможные, как правило не крупные или не сложные элементы, из которых собираются интерфейсы, как правило повторяются чаще обычных компонентов.

Интерфейсные компоненты должны иметь префикс `Ui`.

## Структура компонента

Компонент находится в своей директории, имя которой совпадает с именем компонента, указывается в PascalCase.

Обязательно присутствие одноименного файла с расширением `.astro` - это основной шаблон компонента.

При необходимости можно части компонента вынести в отдельные файлы, которые должны начинаться с имени компонента. В идеале они должны использоваться только в том компоненте, частью которого они являются. Если возникает необходимость использовать их где-то еще - то почти всегда это самостоятельный компонент. Возможен вариант, когда они используются в других компонентах вместе с основным компонентом в его слотах.

При необходимости можно создать одноименный файл с расширением `.scss`, в котором будут храниться стили компонента. Все scss-файлы в директории `src/components` и ее поддиректориях (кроме файлов с именем `tokens.scss`) автоматически подключаются в файле точки сбора стилевых файлов - импортировать их никуда не надо.

Если возникает необходимость сделать кастомные переменные компонента доступными для всего проекта - их надо записать в миксин в файл `tokens.scss`, а сам миксин использовать в файле `src/styles/global/tokens/index.scss`.
Такая необходимость может возникнуть например в компоненте контейнер - когда на мобильных экранах часть элементов надо сверстать "от края до края" - удобно использовать внешние отрицательные отступы, равные внутренним отступам контейнера.

Можно использовать любые файлы с расширением `.js`, для написания логики компонента - но рекомендуется придерживаться нейминга, аналогичного файлам шаблона.

Для удобства исппользования импортов в компоненте используется файл `index.js` - строго для реэкспортов - тогда при импорте достаточно указать только папку компонента. Так же этот файл позволяет контролировать, что будет доступно при импорте других компонентов, правда этот запрет всегда можно обойти, прописав полностью путь к любому файлу.

## Файлы стилей

Кроме стилей компонентов в шаблоне имеются общие стили - они находятся в папке `src/styles`. Точка сбора файл `src/styles/index.scss` - в нем не должно быть никаких селекторов - только подключение других файлов.

Общие стили находятся в папке `src/styles/global`. По названию папок или файлов понятно, что внутри. Если используется группа файлов в папке (например токены) - используется локальная точка сбора - `index.scss`. Тут важно правильно подключать файлы - через директивы `@use` или `@forward`. Устаревшая директива `@import` не используется.

### Миксины

Все миксины файлы с миксинами должны быть прокинуты директивой `@forward` в файл `src/styles/global/mixin/index.scss` - для того, чтобы всегда миксины подключать из одного файла - `@use "@styles/mixin";`

#### Брейкпоинты

Миксин для брейкпоинтов находится в файле `src/styles/global/mixin/breakpoints.scss`. В словаре указывается список допустимых брейкпоинтов, где ключ - имя брейкпоинта, а значение - допустимое в css значение максимальной (или минимальной) ширины (например в пикселях).

При попытки использовать значение не из словаря миксин выкинет ошибку.

По-умолчанию в шаблоне имена брейкпоинтов совпадают со значениями в пикселях - но можно использовать любые удобные имена - например размеры "в майках" или текстовые константы типа `mobile`, `tablet`, `desktop` и т.п.

По-умолчанию используется принцип desktop-first - в нем от указанной ширины брейкпоинта отнимается один пиксель. Есть закомментированная заготовка для mobile-first, в ней ничего не отнимается. Почему так - потому-что это наиболее часто встречающийся подход. При необходимости всегда можно переписать логику миксина под свои нужды.

Рекомендуется не смешивать desktop-first и mobile-first - это делает код запутанным и трудно поддерживаемым. Так же стоит избегать миксинов типа `tablet-only` - код будет чище и понятнее, если все миксины брейкпоинтов будут применяться в одном направлении - просто в следующем брейкпоинте надо отменить или переопределить нужные правила. В таких случаях почти всегда удобно и предпочтительно использовать значение `unset`.
